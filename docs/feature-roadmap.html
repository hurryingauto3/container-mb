Tier 1 Read-Only Additions No design-philosophy change · stays fully non-mutating
Images Tab
image list --format json image inspect <name>
  • Fourth segment in the existing Containers / Volumes / Networks control — no new navigation pattern required.
  • List row: image name + tag, compressed size on disk, short digest (12 chars), platform (os/arch), age since creation.
  • Detail panel on select (lazy image inspect): full digest, exposed ports from config, entrypoint, env vars, layer count. Same master-detail layout already built for containers.
Low effort New tab
Container Logs Viewer
logs <id> -n 200 logs <id> --boot
  • Collapsible "Logs" section below the stats grid in the container detail panel, fetched on demand when expanded.
  • Monospaced, selectable, horizontally scrollable output — last 200 lines. Copy-all button in the section header.
  • --boot toggle switches to VM-level init output, surfacing kernel and vminit messages. Critical for diagnosing startup failures that produce no container-level logs.
Medium effort Panel extension
Disk Usage Widget
system df --format json
  • An additional row in the header summary strip, polled at background frequency (no stats-level cost).
  • Shows images, containers, and volumes disk usage as human-readable sizes — the most useful number to expose without opening a shell.
  • Optional: small proportional bars per type so relative weight reads at a glance.
Low effort Header widget
Volume Detail Inspect
volume inspect <name>
  • Current volume panel shows driver, format, size, and source from the list response. volume inspect adds the full mount path, driver options, and labels.
  • Fetch lazily on volume selection — same pattern as container inspect in PollingCoordinator. No polling overhead.
Low effort Panel extension
Network Detail Inspect
network inspect <name>
  • Current network panel shows mode, subnet, gateway, plugin. network inspect adds the set of attached container IDs and interface-level options.
  • Attached container list can act as a cross-tab deep-link: clicking a container ID in the network panel selects it in the Containers tab.
Low effort Panel extension
Tier 2 System Visibility Still read-only · exposes system-layer state not currently surfaced
Per-Component Version Map
system version --format json
  • Replaces the current flat --version string with a structured breakdown: CLI, runtime, kernel, and vminit component versions.
  • Surfaced in the system status header area — useful for bug reports and confirming upgrade state.
  • system version is more reliable than top-level --version as an API — it returns JSON with named fields rather than a single parseable string.
Low effort Header widget
Builder Status Indicator
builder status --format json
  • A status chip in the header row alongside the existing service-running indicator. Shows whether the build daemon is active — relevant for teams using container build alongside the runtime.
  • Polled at background frequency. Two states: running (with resource config) and stopped. No new UI patterns.
Low effort Header widget
Registry Status
registry list
  • A panel or section in the system info area listing authenticated registries — hostname and auth status.
  • Immediately useful when debugging image pull failures: confirms whether a credential is present before checking network or image reference issues.
  • Low-frequency poll (background only); very cheap call.
Low effort New panel
Machine Management Tab
machine list --format json machine inspect [<id>]
  • Persistent Linux VMs (introduced in v1.0) are a distinct resource type from ephemeral containers — they warrant their own tab, not a section within Containers.
  • List row: VM name, state, CPU and memory config, OS, default flag. Detail panel: full resource config, home mount mode, kernel in use.
  • Medium effort because ContainerJSONMapper needs a new mapping path for the machine shape, and the concept of a "default machine" needs a UI affordance.
Medium effort New tab
System DNS Mappings
system dns list
  • Custom hostname → IP mappings configured via container system dns create. Not visible anywhere in the current UI.
  • Surface as a section within the Networks tab (below the network list) or in the system info panel. Shows domain and resolved address.
  • Useful context when a container is publishing on a custom local domain — clarifies why container.local resolves but an ad-hoc domain doesn't.
Low effort Panel extension
Tier 3 Action Features Requires a deliberate product decision before any of these ship
Design decision required. The app's read-only constraint is a documented philosophy, not an implementation accident — it's in the README, the CLAUDE.md, and the code comments. Adding any action feature ends that guarantee and changes what the app is. The choice isn't "should we add stop?" — it's "are we building a control plane or a monitor?" Settle that first, then the four features below follow naturally.
Container Lifecycle Controls
stop <id> start <id> kill <id> delete <id>
  • Stop, restart, kill, and remove buttons in the container detail panel — contextual to state (only "Start" shown for stopped containers, etc.).
  • Requires: confirmation sheet for destructive operations (kill, delete), post-action forced refresh, error propagation to UI rather than silent failure.
  • The most-requested class of feature for a container monitor. Also the one with the highest blast radius if something goes wrong.
High effort Action
Batch Prune Sheet
prune image prune volume prune network prune
  • A "Clean up" sheet accessible from the summary bar. Shows how much space would be reclaimed per resource type (dry-run first), then a confirm button to execute.
  • Lower stakes than lifecycle controls — prune only removes stopped/unused resources — but still irreversible, so dry-run preview is essential.
  • The disk usage widget from Tier 1 pairs naturally with this: the number motivates the action.
Medium effort Action
Live Log Streaming
logs <id> --follow
  • Real-time log tail in the container detail panel — extends the static Logs Viewer from Tier 1.
  • Requires managing a long-running subprocess that doesn't fit the current request-response ProcessRunner model. Needs: line buffering, max-line cap with auto-scroll, pause/resume, and clean teardown when the container is deselected.
  • Highest engineering effort of any feature on this list. Correct approach: a separate LogStreamingService actor with its own lifecycle, not a patch on ProcessRunner.
High effort Action
Image Pull & Registry Login
image pull <ref> registry login <server> registry logout <registry>
  • Pull-by-reference sheet from the Images tab; progress output streamed via the same infrastructure as log streaming.
  • Registry login requires a credential entry form — password must go to Keychain, not be passed on the command line. --password-stdin via stdin pipe is the correct approach.
  • Scope creep risk: this is the first step toward being a full container UI, which may not be the goal.
High effort Action
UX Improvements No New CLI Calls Client-side or OS API only · quick wins with high perceived polish
Container Age & Uptime
createdAt / startedAt (already parsed)
  • createdAt and startedAt are already mapped in ContainerSummary but not displayed in the list row or detail panel.
  • Show "up 3h" for running containers and "stopped 2d ago" for exited ones directly in the list row, replacing or augmenting the current state badge. Formatted relative to PollingCoordinator's last snapshot time.
Low effort UX only
Search & Filter
(client-side)
  • A search field at the top of each list panel filtering by name, image, or state — live filtering against the in-memory snapshot, no additional CLI calls.
  • State filter buttons (Running / Stopped / All) as a secondary control below the search field.
  • Especially valuable for the Images tab where a large image store is hard to navigate without filter.
Low effort UX only
Sort Controls
(client-side)
  • Column-header taps to sort the container list by CPU %, memory, name, state, or age. Default: state (running first), then name.
  • Sort state persists across polls — the list doesn't jump on refresh, it re-sorts the new snapshot with the current key.
Low effort UX only
State-Change Notifications
UNUserNotificationCenter
  • Notify when a previously-running container transitions to stopped/exited unexpectedly — detected by comparing successive snapshots in PollingCoordinator.
  • Also: service goes down (currently surfaced only as a stale indicator), and optionally when a new container appears.
  • Notification permission is gated; notifications should be opt-in via a Preferences pane or first-launch prompt rather than requested immediately at launch.
Medium effort UX only
